# スキーママスタ（列定義の正本）

**本ファイルが列定義の正本です。**  
新しい列の追加・型・意味の変更は、**必ず本ファイルを先に更新**し、その後スプレッドシート／AppSheet／GAS に反映してください。

Google Drive 上の資料は **参考・現物資料** とし、列の最終決定は本ファイルおよび各 `docs/*.md` に従う。

## 共通ルール

- 論理名（日本語）と物理名（推奨英字スネークケースまたはシート事情に合わせた列名）を対応させる。
- **運行予定ID** は **10_配車予定**（本ドキュメント「B.」）を起点とし、他テーブルから参照できるようにする。
- **運転者ID・車両ID・案件ID・取引先ID** は共通マスタのキーと一致させる。
- 「単独利用OK」の列は、連携しない運用でも NULL 可または任意入力可とすることを許容する設計とする。

## ID・採番ルール

- 各テーブルの**内部Key**は AppSheet の **UNIQUEID()** を基本とする。
- 内部Keyは機械的な一意識別子として使い、**画面表示・帳票表示には原則使わない**。
- 人が見るための番号は **「表示用番号」** 列として別に持つ。
- 表示用番号は必要に応じて **GAS** または **AppSheet 式**で生成する。
- **初期段階では内部Keyのみ必須**とし、表示用番号は後続フェーズで整備する。
- 例：
  - **運行予定ID**：UNIQUEID()
  - **運行予定番号**：`YP-YYYYMMDD-001`
  - **点呼ID**：UNIQUEID()
  - **点呼番号**：`TK-YYYYMMDD-001`
  - **日報ID**：UNIQUEID()
  - **日報番号**：`DR-YYYYMMDD-001`
  - **売上ID**：UNIQUEID()
  - **売上番号**：`SL-YYYYMM-001`

## B. 配車予定表（上流）／ **10_配車予定**

本節 **B. 配車予定表** が定義するのは、印刷用の月別配車表そのものではなく、**月別配車表を変換して作成するシステム連携用DB**（実体シート名の例：**10_配車予定**）である。

- **月別配車表**（`配車予定表_2026年マスター.xlsx` 等）は**横持ち**形式のまま運用し、**システム連携用の列を増やしすぎない**。
- **10_配車予定** は**縦持ち**とし、**1行＝1日・1案件・1担当者・1車両**とする。
- **Googleカレンダー**、**LINE通知**、**点呼記録簿**、**運行日報**、**売上管理表**は、この **10_配車予定** を参照・起点に連携する（初版は**変換処理の実行**により 10_配車予定 を更新する運用を前提とする）。
- **月別横持ち配車表 → 本節の行**への詳細ルールは **`docs/dispatch_conversion_design.md`** にまとめる（読み順・UPSERTキー・空欄・照合状態・変更時の整理など）。

変換元との突合・監査のため、次の列を持つ。

## D. 運行日報 — ヘッダ

1日単位・1運転者（または1車両）など、運用単位は AppSheet で定義。**便ごとの実績**は **運行明細** に格納する。

### 運行日報における運行予定ID（ヘッダ／明細）

- **日報ヘッダの運行予定ID**：その日の日報全体が「どの **10_配車予定** を代表として」起票されたかを示す（親のコンテキスト）。
- **運行明細の運行予定ID**：**便ごとの実績**が、どの **10_配車予定** 行に対応するかを示す。
- **1日・1予定・便がそれに準ずる場合**：ヘッダと明細の運行予定IDは同一になることが多い。
- **1日に複数予定・複数便がある場合**：**実績集計や売上連携では運行明細側の運行予定IDを優先**する（ヘッダは「代表」のみを示しうる）。
- **AppSheet**：明細作成時は親ヘッダの運行予定IDを**初期値として引き継ぐ**ことができる一方、必要に応じ**明細側で別の運行予定IDを選択・変更可能**とする。
