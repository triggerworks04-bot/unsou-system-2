# 運行日報 設計

## 1. 目的

日々の **運行実績** を記録する。論理モデルは `schema_master.md` の **D. 日報ヘッダ** と **E. 運行明細** を正とする。紙／PDF様式は `運送日報フォーマット.pdf`・記入例を参照する。

## 2. ヘッダと明細の分担

| レイヤ | 役割 |
|--------|------|
| **日報ヘッダ** | 対象日・運転者・車両・**出／帰庫**・ODO・**総走行距離**・締め状態など、**日単位の枠** |
| **運行明細** | **便番号**、**実出発時刻**、**実到着時刻**、**出発地**、**目的地**、作業内容、区間 ODO／走行距離等 |

レビュー方針どおり、**実出発・実到着・便・区間地表示は明細側** に置き、ヘッダに載せない。

## 3. 運行予定ID（ヘッダと明細）

- **日報ヘッダの運行予定ID**：その日の日報が「**どの代表の 10_配車予定 行から**」起票されたかを示す。
- **運行明細の運行予定ID**：**便ごと**の実績が「**どの 10_配車予定 に対応するか**」を示す。
- **1日・1予定**に収まる運用では、ヘッダと明細の運行予定IDは**同一**になりやすい。
- **1日に複数予定・複数便**では、**実績集計・売上連携では明細の運行予定IDを優先**する（詳細は `schema_master.md` の同タイトル節）。
- **AppSheet**：明細新規行では親ヘッダの運行予定IDを**初期値**として引き継ぎ、必要に応じ**明細で別予定IDに変更可能**とする。

## 4. 単独利用と連携

- **単独**: 運行予定IDなしでも日報ヘッダの日報日＋運転者＋車両で運用可能。
- **連携**: ヘッダまたは明細に **運行予定ID（Ref）** を入れ、**10_配車予定** との突合や売上計上根拠に使う。

## 5. 参照資料とのマッピング

| 参照元 | メモ |
|--------|------|
| 運送日報フォーマット.pdf | 画面上のグルーピング・必須表示。ヘッダ／明細の画面分割と対応させる |
| 運送日報_記入例.pdf | バリデーション・プレースホルダ |
| 08_11_運送日報入力雛形 | 列取り込み時の一次ソース |
| 運送日報集計 | 集計キーを売上（`運行日報ID` Ref など）と整合 |

## 6. AppSheet

- ヘッダ1件に **関連する運行明細** を複数行で編集できる UI（親子）を前提。
- **日報ID**：UNIQUEID、**日報番号**：表示用（後続整備）。
