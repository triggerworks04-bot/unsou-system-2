# LINE通知 設計（補助機能）

## 1. 目的

運転者などへ **翌日運行予定** 等を通知し、認識齟齬を減らす。**10_配車予定**・カレンダー連携と整合し、**重複送信を防ぐ**ことを最優先の制約とする。印刷用の**月別横持ち配車表**からは LINE 本文を組み立てない。

## 2. 全体での位置づけ

運行システム2は **配車／点呼／日報／売上** を主業務領域とし、LINE通知は **補助機能** としてスプレッドシート上のキューと GAS で実装する。設計の正本は **`docs/`**（本ファイルおよび `schema_master.md` の G. LINE通知キュー）。

Google Drive の資料は引き続き参考扱いとし、送信文言・運用細部は現場確認のうえ本ドキュメントを更新する。

## 3. 送信対象

- **主**：**翌日**の **10_配車予定** から抽出した運行予定（**確定**等、送信条件は運用で定義）。運転者マスタまたは通知先マスタから送信先を解決。
- **補助**：運行管理者向けログ・失敗時アラート等は別途検討。

※ 送信先の特定方法（運転者ID → LINEユーザーID／グループIDのマッピング）は実装フェーズで **個人情報保護** に注意して定義する。

## 4. 送信タイミング

- **例**：毎日定時（例: 18:00）に**翌日分の 10_配車予定**を抽出し、運行一覧文を生成してキューに投入、または一括送信。
- **10_配車予定** の **ステータス**（予定／確定／変更／キャンセル）と、**LINE送信済みフラグ**（**10_配車予定** 上、または **LINE通知キュー** 側の送信済みで代替）により、送る／送らないを制御する。

## 5. データソースと文面形式

- **LINE通知**は、印刷用の月別配車表ではなく **10_配車予定** を元に生成する。
- **翌日分**の **10_配車予定** を抽出し、**運行一覧文**を組み立てる。

**文面の基本形**：

```
【明日の運行予定】

■ {案件名}
{運転者名} / {車両呼称} / {出発地} {出発予定時刻}～
```

- **案件名**・**運転者名**・**車両呼称**は、照合前は **案件名_入力値**・**運転者名_入力値**・**車両呼称_入力値**、照合後はマスタ表示名でもよい（運用で統一）。
- **LINE通知キュー**には、**10_配車予定** から生成した **送信本文** を登録する。
- **LINE送信済みフラグ**は **10_配車予定** 上で持つか、**LINE通知キュー** 側の **送信済みフラグ** で管理するかを運用で選ぶ（**二重管理する場合は更新タイミングを GAS で揃える**）。

## 5a. 送信内容テンプレート（旧案・参考）

以下はプレースホルダ例。上記「基本形」を正とし、必要に応じ本節を併用する。

```
【運行予定】{日付}
{運行予定番号}
{出発地} → {目的地}
出発 {出発予定時刻} ／ 車両 {車両呼称}
案件：{案件名}
※内容は配車確定後の情報です。変更がある場合は管理者に連絡ください。
```

## 6. LINE通知キューの考え方

- スプレッドシート（または同DB）に **1行＝1送信ジョブ** を蓄積する。
- GAS（時間主導トリガまたは手動）が **未送信** かつ **送信予定日時 <= 現在** の行を処理する。
- 送信成功後に **送信済みフラグ**・**送信日時**・**送信結果** を更新する。
- **10_配車予定** の **LINE送信済みフラグ** を同じトランザクションに近い単位で更新する運用を採用する場合、二重経路でも重複しにくくする（実装詳細は GAS 設計に委ねる）。

### 想定列（正本は `schema_master.md` G. LINE通知キュー）

| 列 | 用途 |
|----|------|
| LINE通知ID | 内部Key（UNIQUEID） |
| 通知種別 | 翌日予定／リマインド等 |
| 対象日 | 運行日 |
| 運行予定ID | 紐づけ |
| 運転者ID | 紐づけ |
| 送信先名 | ログ・運用確認用 |
| 送信本文 | 実際に送るテキスト |
| 送信予定日時 | スケジューリング |
| 送信済みフラグ | 未送信／送信済 |
| 送信日時 | 実送信時刻 |
| 送信結果 | 成功／失敗等 |
| エラーメッセージ | 再試行・調査用 |
| 作成日時 / 更新日時 | 監査 |

## 7. 重複送信防止

- **10_配車予定** の `LINE送信済みフラグ` で「この予定行は既に通知済み」を表す**運用**が可能。
- **LINE通知キュー** の `送信済みフラグ` で「このジョブは処理済み」を表す。
- **どちらか一方**、または**両方**で二重防止する場合は、GAS で送信成功時に**同じトランザクションに近い単位**で整合させる。
- キュー投入ロジックは **同一運行予定ID×同一対象日×同一通知種別** で未送信ジョブが複数無いようにするか、処理側でべき等にする。

## 8. GAS で送信済みフラグを更新する方針

1. メッセージ送信 API 呼び出し成功を確認後、`LINE通知キュー` の当該行を更新する。
2. 続いて（または同一処理内で）関連する **10_配車予定** 行の `LINE送信済みフラグ` を `TRUE` に更新する**運用**を採用する場合は、`更新日時` を更新する。
3. 失敗時は `送信結果`・`エラーメッセージ` を記録し、**10_配車予定** のフラグまたはキュー済みフラグは原則立てない（再送可能な運用または手動是正）。
